## 1. Project Overview

Interactive explorer for 2.99M arXiv papers. Svelte 5 static site, precomputed data, browser-side search via sql.js FTS5.

**Architecture**:
```
Data Pipeline (Python/Polars/locally on MacBook)
  → HuggingFace open-index/open-arxiv → 419 monthly Parquet shards
  → Deduplicate, strip embedding vectors
  → Output artifacts to static/data/*
  → git commit → auto-deploy

SvelteKit Static Site (Svelte 5 + Tailwind v4 + sql.js + D3.js)
  └── Deployed to Cloudflare Pages (free tier)
```

**Stack**: Python 3.12+, uv, Polars, SQLite, Svelte 5, SvelteKit, Tailwind CSS v4, D3.js v7, sql.js (sqlite3 compiled to WASM), TypeScript.

**Data source**: `open-index/open-arxiv` on HuggingFace — monthly Parquet shards, each ~2 weeks of new arXiv submissions. Latest shard is incremental; full re-index loads all 419 shards (July 1991 – March 2026).

## 3. Setup & Build Commands

### Prerequisites

```bash
# Python (data pipeline)
brew install uv  # or pip install uv
uv venv
source .venv/bin/activate

# Node (frontend)
npm install      # or pnpm install
```

### Data Pipeline

```bash
# Full build (all 419 shards, 2.99M papers) — ~15 min
uv run python scripts/build_data.py --no-incremental

# Incremental (new shards only) — ~3 min
uv run python scripts/build_data.py

# Sample (for testing frontend) — ~30s
uv run python scripts/build_data.py --sample 10000
```

What the script does:
1. `snapshot_download` from HuggingFace `open-index/open-arxiv` (cached locally after first download)
2. Loads each Parquet shard, strips `vector` column (~8GB per shard)
3. Concatenates, deduplicates by `id`
4. Builds: category graph, category hierarchy, network stats, author graph, author rankings, FTS5 search DBs (split by 5-year range), monthly timeseries
5. Writes `papers.parquet` for incremental rebuilds

Output directory: `static/data/`

**NOTE**: `search*.db` and `papers.parquet` are gitignored. To commit them for deployment, either:
- Remove from gitignore and commit (not recommended: files are huge)
- Copy/search DB files into the build output via a pre-deploy script
- Include in the CF Pages deployment via path override

### Frontend

```bash
npm run dev          # Dev server at localhost:5173
npm run build        # Production build → build/
npm run preview      # Serve build/ locally via Vite
npm run check        # Type-check with svelte-check
```

## 5. Search Architecture Details

### FTS5 Schema

```sql
-- Default FTS5 (content stored inline, not contentless)
CREATE VIRTUAL TABLE papers_fts USING fts5(
    id UNINDEXED,      -- Stored but not indexed (returned by SELECT)
    title,             -- Indexed + stored (returned by SELECT)
    authors,           -- Indexed + stored (returned by SELECT)
    tokenize='porter unicode61'
);
```

`id UNINDEXED` means the ID is stored in the FTS5 table and returned by `SELECT`, but not tokenized for search. `title` and `authors` are both indexed and stored.

### Multi-DB Loading (`src/lib/utils/db.ts`)

```typescript
DB_RANGES = ["1991-1999", "2000-2009", "2010-2014", "2015-2019", "2020-2026"]

// Load a specific range's DB (fetches from /data/search_{range}.db)
loadDb(range: YearRange): Promise<SQL.Database>

// List loaded ranges
loadedRanges(): YearRange[]

// Drop a DB from cache (free memory)
dropDb(range: YearRange): void

// Search across loaded ranges
searchPapers(query, ranges?, limit?, offset?): {results: PaperResult[], total: number}
```

**IMPORTANT**: `searchPapers` does NOT merge results across ranges — it returns results from each range sequentially, then slices to `limit`. This is buggy for multi-range searches (page 2 may skip results from the first range). See "Known Issues" below.

### sql.js Memory

sql.js allocates a WASM heap that grows as DB pages are accessed. Each loaded DB's heap is proportional to the number of pages touched during queries. For a typical search query, ~1-5MB of index pages are accessed. The full DB is NOT loaded into memory.

`dropDb(range)` closes the sql.js database and frees its memory. Useful when user switches year range and no longer needs the old DB.

## 7. Remaining Tasks

### Task 5: Paper Detail Page (`/papers/[id]`)

**Route**: `src/routes/papers/[id]/+page.svelte`

**What to build**:
- Load paper metadata from papers.parquet (or better: build a detail JSON index at build time)
- Show: title, authors, abstract, categories, date, DOI, license
- OpenAlex enrichment: fetch citation count, concepts, related works
- Category breadcrumbs → link to `/papers?category=...`
- BibTeX export button (client-side generated)
- 404 state when ID not found

**Key design decisions**:
- The search DB doesn't store abstracts. Two approaches:
  - A: Build a `papers_detail.db` at build time mapping id→{abstract, categories, doi, ...}
  - B: Load from `papers.parquet` (1.5GB) — not feasible in browser
  - C: Embed a tiny JSON with abstracts for just the papers shown in search results
  - **Recommend**: Build a compact `papers_detail.json` at build time (id→{abstract, categories, doi}) — estimated 200-300MB for 3M papers, or split by year like the search DBs

**Actually simplest approach**: Extend `build_search_db` to also build `papers_detail.db` — a plain SQLite table (no FTS) with just `id TEXT PRIMARY KEY, abstract TEXT, categories TEXT, doi TEXT, license TEXT, update_date TEXT`. This is the same approach that was attempted with `papers_meta.db` before being removed.

```python
# In build_data.py, after build_search_db:
detail_path = out_dir / "papers_detail.db"
mconn = sqlite3.connect(str(detail_path))
mconn.execute("PRAGMA synchronous = OFF; PRAGMA journal_mode = MEMORY")
mconn.execute("CREATE TABLE papers(id TEXT PRIMARY KEY, abstract TEXT, categories TEXT, doi TEXT, license TEXT, update_date TEXT)")
# Batch insert from df
mconn.close()
```

**Detail page load flow**: Load `papers_detail.db` (or the correct year chunk), search for id → render.

**Fallback**: If the detail DB isn't loaded yet, the page can show a loading indicator and load just the relevant ID's metadata via a separate fetch. Since it's static, we could also generate individual JSON files per paper during build: `static/data/papers/2301/10832.json` — but that's 2.99M files.

**Recommended approach**: Use a detail DB split by year (same as search DBs), load on demand when user navigates to paper detail page. The DB only needs `id TEXT PRIMARY KEY, abstract TEXT, categories TEXT`. Abstract is the bulk (~300 bytes avg × 3M = ~900MB if all in one file, or ~200MB per year split). Since detail is loaded after search (user has already found the paper), they can wait for a small download.

### Task 6: Author Network Page (`/authors`)

**Route**: `src/routes/authors/+page.svelte` (currently stub)

**Data**: `static/data/author_graph.json` (50K nodes, 200K edges) + `static/data/author_rankings.json` (top 1000)

**What to build**:
- D3.js force-directed graph with Canvas renderer (50K nodes → SVG will choke)
- Node size = paper count (from `weight`), color = community (optional)
- Top authors list (from rankings)
- Search/filter input → filter displayed nodes by name
- Click node → navigate to `/authors/[id]`
- Zoom/pan with mouse and touch
- Hover tooltip with author name and paper count

**Key patterns from CategoryGraph.svelte to follow**:
- Same D3 force layout pattern (`d3.forceSimulation`, `forceManyBody`, `forceLink`, `forceCenter`)
- Same loading/error state handling
- Same `$effect` pattern for running D3 when DOM is ready

**Canvas vs SVG decision**: Use Canvas for this graph. 50K circles × 200K lines in SVG creates DOM nodes the browser can't scroll through. Canvas renders directly to a bitmap.

```typescript
// Canvas rendering pattern for author graph
function renderCanvas(graph: AuthorGraphData, canvas: HTMLCanvasElement) {
  const ctx = canvas.getContext("2d")!;
  const { width, height } = canvas;
  const nodeMap = new Map(graph.nodes.map(n => [n.id, n]));

  const simulation = d3.forceSimulation(graph.nodes as any)
    .force("link", d3.forceLink(graph.edges).id((d: any) => d.id))
    .force("charge", d3.forceManyBody().strength(-10))
    .force("center", d3.forceCenter(width/2, height/2));

  simulation.on("tick", () => {
    ctx.clearRect(0, 0, width, height);
    // Draw edges (thin lines)
    graph.edges.forEach(e => {
      const s = e.source as any, t = e.target as any;
      ctx.beginPath(); ctx.moveTo(s.x, s.y); ctx.lineTo(t.x, t.y);
      ctx.strokeStyle = "rgba(100,100,100,0.1)"; ctx.stroke();
    });
    // Draw nodes (circles)
    graph.nodes.forEach(n => {
      const d = n as any;
      ctx.beginPath(); ctx.arc(d.x, d.y, Math.sqrt(n.weight)/5, 0, Math.PI*2);
      ctx.fillStyle = "#4a90d9"; ctx.fill();
    });
  });
}
```

### Task 11: Search Results Bug Fix

**Problem**: `searchPapers()` in `db.ts` currently searches multiple DBs but returns results sequentially, then slices to `limit`. This means multi-range search pagination is broken — page 2 might skip the last results from the first DB.

**Expected behavior**: Merge results by FTS5 rank across all DBs, then paginate.

**Fix**:

```typescript
export function searchPapers(
  query: string,
  ranges?: YearRange[],
  limit = 30,
  offset = 0,
): { results: PaperResult[]; total: number } {
  const q = sanitizeQuery(query);
  if (!q) return { results: [], total: 0 };

  const targets = ranges ?? [...dbCache.keys()];
  if (targets.length === 0) return { results: [], total: 0 };

  let total = 0;
  const allRows: Array<{ rank: number; range: string; rowid: number; id: string; title: string; authors: string }> = [];

  for (const range of targets) {
    const db = dbCache.get(range);
    if (!db) continue;

    // Get total for this range
    const countStmt = db.prepare("SELECT COUNT(*) as cnt FROM papers_fts WHERE papers_fts MATCH ?");
    countStmt.bind([q]);
    if (countStmt.step()) total += countStmt.getAsObject().cnt;
    countStmt.free();

    // Get ALL matching rowids + table-level rank (can't order across DBs)
    const stmt = db.prepare("SELECT rowid, id, title, authors, rank FROM papers_fts WHERE papers_fts MATCH ? ORDER BY rank");
    stmt.bind([q]);
    while (stmt.step()) {
      const r = stmt.getAsObject() as any;
      allRows.push({ rank: r.rank, range, rowid: r.rowid, id: r.id, title: r.title, authors: r.authors });
    }
    stmt.free();
  }

  // Sort by rank across all DBs (lower rank = better match in FTS5)
  allRows.sort((a, b) => a.rank - b.rank);

  const sliced = allRows.slice(offset, offset + limit);
  return {
    results: sliced.map(r => ({ id: r.id, title: r.title, authors: r.authors })),
    total,
  };
}
```

## 8. Key Decisions & Conventions

1. **FTS5 default (not contentless)** — `CREATE VIRTUAL TABLE papers_fts USING fts5(id UNINDEXED, title, authors, ...)` — content stored inline, `SELECT` returns all columns directly without join.

2. **DB split by 5-year ranges** — Each file under 500MB. Frontend loads on demand.

3. **Static site, no server** — All data precomputed. No API routes. No serverless functions. `prerender = true` on every route.

4. **`separators=(",", ":")` on JSON writes** — No whitespace. Minimal file size.

5. **Polars for data processing** — Fast, lazy, memory-efficient. No Pandas.

6. **`uv run python script.py`** — Always prefix with `uv run` to use the venv.

7. **Domain colors from `DOMAIN_COLORS`** — Hardcoded per top-level domain. `DOMAIN_NAMES` has fewer entries than `DOMAIN_COLORS` — tight subdomains (gr-qc, quant-ph, hep-th) exist only in colors dict.

8. **Category aliases** — See `CATEGORY_ALIASES` dict. Some old arXiv category names are mapped to modern equivalents (e.g., `cmp-lg` → `cs.CL`).

9. **Component style**: Svelte 5 runes (`$state`, `$derived`, `$effect`), no stores for basic state, TypeScript. Comments are minimal per project convention.

10. **Tailwind v4** — Uses `@import "tailwindcss"` (not the v3 `@tailwind` directives). Dark mode via `.dark` class on `<html>`. Custom dark variant: `@variant dark (&:where(.dark, .dark *))`.

## 9. Known Issues

| Issue | Severity | Workaround |
|-------|----------|------------|
| Multi-DB search pagination is broken (results not merged correctly) | Medium | See Task 11 fix above |
| `search_2020-2026.db` at 447MB may hit CF Pages 500MB soft limit within ~1 year | Low | Split 2020s into 5-year chunks when needed |
| sql.js WASM heap grows with query volume (no automatic LRU eviction) | Low | Use `dropDb()` to clear ranges no longer needed |
| FTS5 `MATCH` on author names with special characters (periods, hyphens, apostrophes like O'Brien) may give unexpected results | Low | The `unicode61` tokenizer handles most cases; for edge cases, use double-quoted phrases |
| Author disambiguation is string-based (no ORCID) — "John Smith" could be multiple people | Low | Future improvement: cluster by name + co-author graph |
| No detail page data for paper IDs (no abstract/Categories accessible from browser) | High (blocks Task 5) | Build `papers_detail.db` with `id` → `{abstract, categories, doi, ...}` |
