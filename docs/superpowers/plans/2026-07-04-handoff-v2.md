## 1. What changed and why

The old plan deployed 45–447MB SQLite files as Cloudflare Pages static assets. **That cannot work: CF Pages caps static assets at 25 MiB per file** (the old handoff's "500MB soft limit" table is wrong).

New architecture, already implemented and verified locally:

- The `.db` files live in a **Cloudflare R2 bucket** (free: 10GB storage, zero egress), NOT in the site deploy.
- The frontend uses **`sql.js-httpvfs`** (already in package.json): SQLite compiled to WASM that fetches individual 4KB pages via HTTP Range requests from a Web Worker. A search on the 447MB DB fetches ~11MB instead of 447MB (measured).
- The site itself stays fully static (adapter-static, unchanged) and deploys to CF Pages fine because the DBs are gitignored and never enter the build.

Also fixed in this session:
- `year_from_id` in `scripts/build_data.py` now parses old-style arXiv IDs (`hep-th/9901001`). Before, **all pre-2007 papers were silently excluded from search** and `search_1991-1999.db` was never created.
- Multi-range search pagination bug (old handoff Task 11): `searchPapers()` now merges results across year ranges by FTS5 rank. Task 11 is DONE.
- The pipeline now also builds `detail_{range}.db` files (id → abstract, categories, doi, license, update_date), and `src/lib/utils/db.ts` exports `getPaperDetail(id)` — ready for Task 5. **But read the WARNING in step 4 before uploading detail DBs.**

## 3. Step-by-step: finish the deployment

### Step 3.1 — Rebuild the data artifacts (local, ~20 min)

The existing search DBs were built before the old-style-ID fix and without explicit page_size. They must be rebuilt.

```bash
cd ~/Documents/GitHub/arxiv-data-explorer
rm static/data/search_*.db
uv run python scripts/build_data.py --no-incremental
```

Expected output: five `search_*.db` (now INCLUDING `search_1991-1999.db`) and five `detail_*.db` files, each reporting a size and paper count. Total papers across search DBs should be ~2.99M (previously ~2.57M because pre-2007 papers were dropped).

Verify page size (must print 4096):

```bash
sqlite3 static/data/search_2020-2026.db "PRAGMA page_size"
```

**WARNING**: If you change `SQLITE_PAGE_SIZE` in build_data.py, you MUST change `REQUEST_CHUNK_SIZE` in `src/lib/utils/db.ts` to the same value and rebuild+reupload every DB. They must always match or every query fails.

### Step 3.2 — Create the R2 bucket (user's Cloudflare account required)

```bash
npm install -g wrangler
wrangler login
wrangler r2 bucket create arxiv-explorer-data
```

Then in the Cloudflare dashboard (dash.cloudflare.com → R2 → arxiv-explorer-data → Settings):
1. **Public access** → "Allow Access" via r2.dev subdomain. Note the URL, it looks like `https://pub-<hash>.r2.dev`.
2. **CORS policy** → paste the contents of `infra/r2-cors.json`.

### Step 3.3 — Upload the search DBs

`wrangler r2 object put` fails on files over ~300MB, so the upload script uses rclone:

```bash
brew install rclone
```

Create an R2 API token: dashboard → R2 → Manage R2 API Tokens → Create → "Object Read & Write". Put the credentials in `~/.config/rclone/rclone.conf`:

```ini
[r2]
type = s3
provider = Cloudflare
access_key_id = <token key id>
secret_access_key = <token secret>
endpoint = https://<account-id>.r2.cloudflarestorage.com
```

Upload search DBs only (see WARNING in step 4 about detail DBs):

```bash
for f in static/data/search_*.db; do
  rclone copyto "$f" "r2:arxiv-explorer-data/$(basename $f)" --s3-upload-cutoff 100M --s3-chunk-size 100M --progress
done
```

(`scripts/upload_r2.sh` does the same but includes detail DBs — don't use it until the detail-DB size question is resolved.)

Verify range requests work (expect `HTTP/2 206` and a `content-range` header):

```bash
curl -s -o /dev/null -D - -r 0-99 https://pub-<hash>.r2.dev/search_2000-2009.db
```

### Step 3.4 — Deploy the site to Cloudflare Pages

No adapter change needed — `adapter-static` output works on Pages.

1. Push the repo to GitHub (DBs are gitignored, so the repo stays small).
2. Cloudflare dashboard → Workers & Pages → Create → Pages → Connect to Git → select the repo.
3. Build command: `npm run build`. Output directory: `build`.
4. **Environment variable** (Settings → Environment variables, Production):
   `VITE_DATA_BASE_URL` = `https://pub-<hash>.r2.dev` (no trailing slash).
5. Deploy.

**WARNING**: `.env.local` (gitignored) sets `VITE_DATA_BASE_URL=http://localhost:8082` for local dev. Never commit it. If production search fails with connection errors to localhost:8082, the env var wasn't set in Pages.

### Step 3.5 — Verify production

- Open `<site>/papers`, search "quantum computing" → results appear within a few seconds.
- Search a pre-2007 paper: query "asymptotic freedom" with year range 1991-1999 → results must appear (proves the old-style-ID fix shipped).
- Browser devtools → Network: requests to `pub-<hash>.r2.dev` return status 206.

## 5. Gotchas encountered (do not rediscover these)

1. **`sql.js-httpvfs` is UMD/CommonJS.** Import it as `import pkg from "sql.js-httpvfs"; const { createDbWorker } = pkg;`. A named import compiles but crashes Vite SSR/prerender with "Named export 'createDbWorker' not found".
2. **Never call `worker.db.query(sql, param1, param2)`.** The Comlink proxy drops variadic params and FTS5 sees an empty match string ("fts5: syntax error near ''"). Always use `worker.db.exec(sql, [params])` via the `dbOf()` facade in db.ts, and map rows with `toObjects()`.
3. **Comlink's `Remote<LazyHttpDatabase>` type loses `exec`/`close`** (sql.js type interop). The `dbOf()` facade in db.ts exists for this; don't fight the types elsewhere.
4. **Local dev needs a Range-capable server for the DB files.** Vite doesn't reliably serve ranges. Use `npx http-server static/data -p 8082 --cors -c-1` (configured in `.claude/launch.json` as "data") together with `.env.local`. If search hangs locally with the whole file downloading, the data server isn't honoring Range headers.
5. **Python sqlite3 auto-opens transactions** — `conn.commit()` is required before `VACUUM` in build_data.py (already in place; keep it if you edit those functions).
6. **`sanitizeQuery` strips non-alphanumeric chars**, so accented author names aren't findable via search. Known pre-existing limitation, fine for v1.

## 6. Cost/limits watch

- R2 reads: ~2,700 range requests per search query (measured on the 447MB DB). Free tier = 10M reads/month ≈ 3,700 queries/month. If the site gets real traffic, raise `SQLITE_PAGE_SIZE`/`REQUEST_CHUNK_SIZE` to 16384 or 32768 (fewer, larger reads; rebuild + reupload everything, see step 3.1 WARNING) and/or drop the exact `COUNT(*)` query in `searchPapers` (it walks full posting lists; replace `total` with a has-more heuristic).
- R2 storage: 818MB of 10GB used after step 3.3.
